# Baseline Strategies and Predictive Models

## Purpose

This module provides the first benchmarking layer for the project. It answers two concrete questions before any deep-learning work:

1. Can simple, interpretable trading rules already isolate better post-cost funding arbitrage opportunities?
2. Can lightweight supervised models improve on those rules using the engineered feature set?

These baselines are meaningful because they are easy to explain, cheap to train, and directly aligned with the project's tradable post-cost labels.

## Inputs

The baseline pipeline consumes the supervised dataset generated by `build-labels`:

- default input: `data/processed/supervised/binance/btcusdt/1h/btcusdt_supervised_dataset.parquet`
- default classification target: `target_is_profitable_24h`
- default regression target: `target_future_net_return_bps_24h`
- default split column: `split`
- default readiness filter: `supervised_ready == 1`

The default 24-hour target is a practical implementation choice. Under the current cost assumptions, positive labels are very sparse. The 24-hour horizon is still difficult, but it is slightly less sparse than the 8-hour label and therefore a better first benchmark target.

## Rule-Based Baselines

The implemented rule-based examples are:

1. `funding_threshold_2bps`
   Signal when `funding_rate_bps >= 2.0`.
2. `spread_zscore_1p5`
   Signal when `spread_zscore_72h >= 1.5`.
3. `combined_funding_spread`
   Signal when both `funding_rate_bps >= 1.0` and `spread_zscore_72h >= 1.5`, with `positive_funding_regime == 1`.

Interpretation:

- positive funding means shorts are paid, which is favorable for a `short perp + long spot` carry trade
- a wide positive spread z-score means the perp looks relatively rich versus spot
- the combined rule checks whether both carry and basis dislocation line up at the same time

These rules are intentionally simple. They establish whether obvious heuristics already capture some useful structure before using learned models.

## Predictive Baselines

The implemented ML baselines are:

1. Logistic regression
   Input: leakage-safe numeric engineered features
   Output: probability of `target_is_profitable_24h == 1`
2. Ridge regression
   Input: leakage-safe numeric engineered features
   Output: predicted `target_future_net_return_bps_24h`
3. Optional random-forest baselines
   Present in config but disabled by default to keep the first run lightweight

Modeling notes:

- only rows with `supervised_ready == 1` are used
- training uses the existing time-series `train` split only
- validation and test are evaluated without refitting
- feature selection excludes `target_*` columns and key metadata columns
- columns with too much missingness or zero variance are dropped before fitting
- linear models use median imputation, and scaling when configured

## Outputs

Default artifact directory:

`data/artifacts/models/baselines/binance/btcusdt/1h/btcusdt_24h_default/`

Key outputs:

- `baseline_predictions.parquet`
  Row-level predictions and rule signals for every enabled baseline.
- `baseline_metrics.parquet`
  Split-level metrics for each model.
- `baseline_leaderboard.parquet`
  Validation/test summary table.
- `baseline_report.md`
  Short markdown summary for presentation and review.
- `feature_columns.json`
  Exact feature list used by predictive models.
- `models/*.joblib`
  Saved scikit-learn model artifacts.
- `diagnostics/*_diagnostics.csv`
  Coefficients or feature importances when available.
- `baseline_manifest.json`
  Config and artifact summary for reproducibility.

## Prediction Table Contract

`baseline_predictions.parquet` is designed so later backtests can consume it directly. Important columns are:

- `timestamp`
- `split`
- `model_name`
- `model_family`
- `task`
- `signal_direction`
- `signal`
- `decision_score`
- `signal_threshold`
- `signal_strength`
- `predicted_probability`
- `predicted_return_bps`
- `actual_label`
- `actual_return_bps`

The intended downstream use is:

- classifiers and rules provide binary trade candidates through `signal`
- regressors provide both `signal` and a ranked expected-return score through `predicted_return_bps`
- backtests can later filter or rank by `signal_strength`, `predicted_probability`, or `predicted_return_bps`

## Evaluation Summary

Classification-style outputs report:

- accuracy
- precision
- recall
- F1
- ROC-AUC when defined
- average precision when defined
- signal count and signal rate
- realized average return of signaled rows
- signal hit rate

Regression outputs report:

- MAE
- RMSE
- R-squared
- Pearson correlation
- directional accuracy
- signal count and signal rate
- realized average return of signaled rows
- signal hit rate

Signal-based metrics are included because post-cost profitable labels are sparse, so pure classification metrics alone can be misleading.

## Caveats

- The current label setup yields extremely sparse post-cost positives in late validation/test periods.
- This is informative for the project because it highlights how hard post-cost funding arbitrage is under realistic frictions.
- The first baseline layer should therefore be treated as a benchmarking and diagnostics tool, not proof that the strategy is already good.
